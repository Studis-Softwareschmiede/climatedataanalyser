# Upgrade-Lauf 2026-06-02 — Lessons (für /retro → Pack/Prozess-Verbesserung)

Quelle: autonomer `/upgrade`-Lauf (java 11→21, spring-boot 2.6→3.3.13 gelandet; SB4 blocked).
Diese Lektionen sind **verallgemeinerbar** und sollten via `/retro` in die globalen Packs / das Upgrade-Subsystem fließen.

## L1 — java-version-Bump + Nicht-UTF-8-Locale (HIGH, generalisierbar)
**Symptom:** Nach `java 11→17` 2 DAO-Tests rot (`Baden-Württemberg`-Query matcht nichts), nur auf JDK 17; master-CI (JDK 11) grün.
**Ursache:** `@Sql`-Seed-Daten mit Umlauten; JDK < 18 leitet `file.encoding` aus der **Platform-Locale** ab. Auf einer Nicht-UTF-8-Locale (z.B. `LANG=C` → US-ASCII) werden die Seeds falsch gelesen. JDK 11-CI lief zufällig UTF-8.
**Fix/Regel:** Bei **jeder** java-version-Bump-Stufe (AC-L*) proaktiv `-Dfile.encoding=UTF-8 -Dsun.jnu.encoding=UTF-8` auf surefire **und** failsafe pinnen → deterministisch auf jeder JDK/Locale. (JDK 18+ macht UTF-8 default via JEP 400 — Pin bleibt für 17 nötig.)
→ **java-Pack** (C-Sektion Floor) + **upgrade-subsystem** „language-rung checklist".

## L2 — OpenRewrite Recipe-Ordering / Chicken-Egg (HIGH)
**Symptom:** `UpgradeSpringBoot_3_3` migriert **nicht** Spring Batch. Der separate `SpringBatch4To5Migration` failte, weil der un-migrierte `JobBuilderFactory`/`StepBuilderFactory`-Code unter Batch 5 nicht kompiliert → OpenRewrite kann nicht type-attributen.
**Regel:** Batch-Recipe **in derselben** `activeRecipes`-Liste wie das Boot-Recipe fahren (komma-separiert), ODER zuerst die Factory-Reste manuell entfernen (Compile herstellen) und **dann** das Batch-Recipe. Generell: framework-major-Recipe + zugehörige Sub-Recipes gemeinsam aktivieren.
→ **upgrade-subsystem** „OpenRewrite-Strategie" + **spring-boot-3-Pack**.

## L3 — JUnit4→5-Migration aktiviert tote Lifecycle-Methoden (MEDIUM)
**Symptom:** OpenRewrite `@Before`→`@BeforeEach` ließ ein `setup()` plötzlich **laufen**, das unter JUnit 5 vorher still ignoriert war (JUnit-4-`@Before`). Das Über-Seeding brach 2 Tests (`1` vs `59`).
**Regel:** Nach JUnit-4→5-Migration **neu-laufende** `@BeforeEach`/`@AfterEach` auditieren — sie können Testdaten/Verhalten ändern, obwohl der Code „nur" annotation-migriert wurde.
→ **reviewer-Checklist** (Test-Migration) + **upgrade-subsystem**.

## L4 — Spring Boot 4 / Batch 6 / Hibernate 7 Breaking Changes (für spring-boot-4-Pack)
- `org.hibernate.Session.saveOrUpdate()` **entfernt** (Hibernate 7) → `merge()`.
- Spring Batch 6: Item-Reader nach `org.springframework.batch.infrastructure.item.*` verschoben; `FlatFileItemReader`/`MultiResourceItemReader` **no-arg-Konstruktor entfernt** → Builder/Ctor mit `LineMapper`.
- **Batch-Reader sind oft nicht unit-getestet** (H2-Suite deckt den File-Import nicht ab) → SB4 braucht **supervised Runtime-Verifikation** vor Merge (gate-integrity: Compile+Unit grün ≠ verifiziert).
→ **spring-boot-4-Pack** (A-Sektion) + **upgrade-subsystem** „untested-runtime → supervised".

## L5 — Prozess: Stepping-Stones funktionieren (LOW)
3.3→3.5→4.0 über aufeinanderfolgende OpenRewrite-Recipes ist sauber; nach **jedem** Major grün verifizieren, bevor der nächste läuft. `-o` (offline) bricht nach Major-Bump (neue Plugin-Versionen nicht gecached) → online bauen.
